# Optimization Backlog

本文档记录后续可以选择实施的优化方向。这里的条目默认不代表当前必须实现，也不代表已经决定实现。

## 使用规则

- 每完成一个外设或功能阶段后，补充可选优化项。
- 优化项只记录方向、收益、代价、触发条件和风险，不在当前功能里顺手实现。
- 只有当优化项被明确选中后，才进入 `PROCESSING.md` 作为当前阶段任务。
- 发生真实性能问题、资源压力、体验问题或维护问题时，优先从本文档选择优化方向，而不是临时堆补丁。

## 状态定义

- `候选`：已经发现可能有价值，但暂不实施。
- `观察中`：需要更多板端现象或数据再决定。
- `计划中`：已决定后续实施，但还未开始。
- `已实施`：已经完成，正式进度见 `PROCESSING.md`。
- `放弃`：确认不适合当前项目。

## OV2640 摄像头

### OV-001：摄像头帧缓冲动态申请

- 状态：候选
- 当前现状：
  - OV RGB565 捕获缓冲是静态全局数组。
  - `240x180` 预览配置下约固定占用 88 KB RAM。
- 优化方向：
  - 进入 `APP ov live` 或拍照流程时用 `rt_malloc()` 申请帧缓冲。
  - 退出摄像头功能时用 `rt_free()` 释放。
- 预期收益：
  - 摄像头不用时释放大块 RAM。
  - 后续 MQTT、识别、缓存、文件操作等功能有更多内存余量。
- 代价：
  - 需要处理申请失败。
  - 需要保证 DCMI/DMA 使用期间缓冲不被释放。
  - 需要注意内存碎片。
- 触发条件：
  - 后续功能出现内存不足、线程创建失败、栈溢出、文件/网络缓冲不足。
  - 需要继续提高摄像头分辨率。
- 风险：
  - 动态内存生命周期管理错误会导致 DMA 写非法地址。

### OV-002：实时预览尺寸档位化

- 状态：观察中
- 当前现状：
  - 已验证隔行刷新比整块刷新流畅。
  - 当前预览尺寸为 `240x180`，需要板端继续观察流畅度和稳定性。
- 优化方向：
  - 提供几个固定档位，例如 `200x150`、`240x180`、`280x210`。
  - 不在运行时暴露复杂参数，先通过宏或单一配置选择。
- 预期收益：
  - 可以在清晰度、画面尺寸、流畅度、RAM 占用之间快速取舍。
- 代价：
  - 需要维护每个档位的缓冲大小、窗口位置和验证记录。
- 触发条件：
  - `240x180` 过小或过卡。
  - 需要在比赛展示时快速切换到更稳的配置。
- 风险：
  - 档位过多会让代码和文档变复杂。

### OV-003：块级差分刷新

- 状态：候选
- 当前现状：
  - 实时预览采用隔行刷新，降低单次 LCD 总线占用。
- 优化方向：
  - 保存上一帧或低成本摘要。
  - 将预览区域分成若干小块，只刷新变化明显的块。
- 预期收益：
  - 静态场景下显著减少 LCD 写入量。
  - 可以降低局部画面变化时的刷新压力。
- 代价：
  - 需要额外上一帧缓冲或块摘要缓冲。
  - 需要比较算法。
  - 动态场景下收益可能很小。
- 触发条件：
  - `240x180` 或更高尺寸仍有明显刷屏感。
  - 场景大部分时间是静态画面。
- 风险：
  - 算法阈值不好会出现局部残影或画面撕裂。
  - 代码复杂度明显上升。

### OV-004：拍照高分辨率、预览低分辨率分离

- 状态：候选
- 当前现状：
  - 实时预览和一次性预览使用同一组尺寸参数。
- 优化方向：
  - 实时预览保持较低分辨率以保证流畅。
  - 点击拍照后停止 live，单独捕获更高分辨率定格图。
- 预期收益：
  - 实时取景流畅。
  - 拍照结果更清晰。
- 代价：
  - 需要两套 OV 输出尺寸配置。
  - 拍照切换时会有短暂停顿。
- 触发条件：
  - 需要保存照片或给后续识别算法使用更清晰图像。
- 风险：
  - 切换传感器寄存器配置可能引入新的时序问题。

## LCD / UI

### LCD-001：摄像头页面布局重做

- 状态：候选
- 当前现状：
  - 摄像头页面框架最初按较大预览区设计。
  - 实际实时预览尺寸会根据流畅度调整。
- 优化方向：
  - 让预览框尺寸和当前预览分辨率匹配。
  - 减少空白区域，让状态栏和拍照按钮更紧凑。
- 预期收益：
  - 视觉比例更协调。
  - 手持查看时更舒服。
- 代价：
  - 需要重新调整触摸区域坐标和状态区域坐标。
- 触发条件：
  - 预览尺寸确定后仍感觉页面突兀。
- 风险：
  - 改布局可能影响已验证的触摸拍照区域。

### LCD-002：LCD 写入性能专项优化

- 状态：候选
- 当前现状：
  - LCD 通过 FSMC/8080 写 GRAM。
  - 大面积实时图像刷新仍受总线写入速度限制。
- 优化方向：
  - 检查 FSMC 写时序是否还有安全提速空间。
  - 评估是否能使用 DMA/FSMC 批量搬运。
  - 优化 RGB565 写循环中的分支和重复计算。
- 预期收益：
  - 所有大面积 LCD 刷新都会变快。
- 代价：
  - 需要严谨验证 LCD 稳定性。
  - FSMC 时序过快可能导致花屏或随机异常。
- 触发条件：
  - 摄像头预览尺寸继续增大。
  - 后续 UI 需要频繁大面积刷新。
- 风险：
  - 显示稳定性下降。
  - 不同 LCD 模块批次容限可能不一致。

## SD / 文件系统

### SD-001：照片保存流程异步化

- 状态：候选
- 当前现状：
  - SD 卡已验证读写。
  - 当前摄像头拍照保存流程已经实现为同步 BMP 写入。
  - 同步写入简单可靠，但保存期间 live 线程会短暂停留。
- 优化方向：
  - 拍照后先把数据放入队列或缓冲。
  - 使用独立线程写 SD 卡，避免阻塞 UI。
- 预期收益：
  - 拍照后 UI 不会长时间卡住。
- 代价：
  - 需要队列、缓冲生命周期和错误回报机制。
- 触发条件：
  - 实现拍照保存到 SD 卡。
- 风险：
  - 掉电或写失败时需要明确错误处理。

## ESP / 网络

### NET-001：ESP8266 网络栈恢复验证

- 状态：候选
- 当前现状：
  - ESP8266 网络验证已暂时推迟。
  - `esp0` 曾出现 `LINK_DOWN`，MQTT 尚未完成。
- 优化方向：
  - 先恢复 at_device 最小联网链路。
  - 再接 SAL/netdev。
  - 最后接 MQTT。
- 预期收益：
  - 避免手写 AT/socket/MQTT 逻辑。
- 代价：
  - 需要重新核对 RT-Thread Settings 和软件包兼容性。
- 触发条件：
  - 摄像头和本地外设阶段稳定后恢复联网。
- 风险：
  - 旧软件包可能与当前 RT-Thread Studio 版本存在兼容问题。
